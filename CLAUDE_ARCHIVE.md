# Mossmark — Key Design Decisions Archive

This file contains the full iteration-by-iteration implementation history from CLAUDE.md's Key Design Decisions section. It is a read-only historical record — not a source of current constraints. For active patterns, see the Key Design Decisions section of [CLAUDE.md](CLAUDE.md).

---

- **Render pipeline & input, locked in at bootstrap**: kept the Unity 6 2D template's URP 2D Renderer (2D lights available for the later day/night cycle) and the new Input System with project-wide actions (see "Input Actions" above), rather than reverting to built-in 2D / legacy `Input.GetAxis` as the old CLAUDE.md (written for P1) specified.
- **`PlayerController`/`CameraFollow`/`TriangleSpriteGenerator`** ported from P1's `PlayerMovement.cs`, `CameraFollow.cs`, `TriangleSpriteGenerator.cs`, dropping the `CombatManager`/`DayCycleManager`/`RestInteractable` hooks that don't exist in P2 yet (per PROTOTYPE2.md, Section 1: "Drop the `CombatManager`/exhaustion-collapse hooks").
- **`AttentionInput` uses the `Attend` action's `started`/`canceled` callbacks, not `performed`.** The Hold interaction's `started` fires on press and `canceled` fires on release regardless of the interaction's own (default 0.4s) duration, so these give clean raw press/release signals. `AttentionManager` owns the actual hold-duration timing per-target via `IAttendable.AttentionDuration`, so the Hold interaction's configured duration is present but unused — left as-is rather than switching to a plain Button interaction, since `started`/`canceled` already give the needed signals.
- **`AttendableZone` resolves `IAttendable` via `GetComponent<IAttendable>()`** on its own GameObject rather than a serialized interface reference — every attendable type is expected to carry its `IAttendable` component and an `AttendableZone` on the same GameObject, which fits the inactive-GO procedural spawn pattern for later iterations.
- **Player's attention-range trigger doubles as its required `Collider2D`**: a `CircleCollider2D` (trigger, radius 1.5) was added directly to the Player in the editor for `AttendableDetector`'s range queries; `PlayerController.Awake()`'s fallback `AddComponent<CircleCollider2D>()` now finds this one and skips adding a second.
- **`AttentionManager` finds `AttendableDetector`/`AttentionInput` via `FindAnyObjectByType` in `Start()`** rather than serialized cross-scene references — keeps the Player's components decoupled from the manager's placement in the hierarchy. (Originally `FindFirstObjectByType`; switched to `FindAnyObjectByType` in Iteration 3 since the former is obsolete and ordering guarantees aren't needed for a one-of-each singleton lookup.)
- **`AttentionManager.AttendingTarget`** exposes the in-progress attend target separately from `CurrentTarget` (the detector's nearest target, which can change mid-hold). `AttendableOverlayUI` reads `AttendingTarget` while `State == Attending` and `CurrentTarget` otherwise, so the overlay always reflects the attendable actually being held.
- **`AttendableOverlayUI` builds its `VisualElement` tree entirely in code** (`OnEnable`, via `UIDocument.rootVisualElement`) and creates a runtime `PanelSettings` via `ScriptableObject.CreateInstance` rather than referencing UXML/USS/PanelSettings assets — keeps the overlay code-first/grey-box and avoids asset-reference wiring through MCP-Unity's `update_component` (which has no reliable way to assign `UnityEngine.Object` asset references). The hold-progress bar reuses P1's `[####....]` text-bar pattern (10 segments, built from `HoldProgress01`).
- **"Overlay UI" GameObject** (root, `UIDocument` + `AttendableOverlayUI`) is a new sibling to `AttentionManager` in the scene — UI reads `AttentionManager.Instance` state each frame in `Update()`, consistent with "UI holds no game logic."
- **`PlaceholderAttendable` removed in Iteration 4**: now that `GenericWildernessSpotAttendable` is a real attendable type, the script, its "Placeholder Attendable" GameObject, and the corresponding scene wiring were all deleted per the removal note this section previously carried.
- **`GenericWildernessSpotAttendable` (Iteration 4) lives in `Mossmark.World`**, not `Mossmark.Attention` or `Mossmark.Development` — it's the first of the "generic spots/POIs" attendable types `Mossmark.World` was reserved for, keeping `Mossmark.Attention` limited to the framework itself (interfaces, zones, detector, manager, overlay).
- **`GenericWildernessSpotAttendable.AttentionDuration` is `0f`**: foraging is an instant yield-roll rather than a timed hold. `AttentionManager`'s existing hold-duration logic handles a zero duration as "complete immediately on attend", so no changes to `AttentionManager`/`AttendableOverlayUI` were needed — the overlay's progress bar simply never has time to render. (Superseded in Iteration 8.5 — `AttentionDuration` is now a non-zero tick interval and foraging repeats while held.)
- **Yield rolls use a weighted pool (`ItemYield[] commonYields`) plus an independent rare-drop roll (`ItemYield rareYield` + `rareDropChance`)**: every attend rolls exactly one common yield by weight, and separately has a flat chance (default 8%) to also award the rare yield. Both rolls are logged via `Debug.Log` for now — there's no toast/notification UI yet.
- **`ItemDatabase` (from PROTOTYPE2.md's P1 reference list) was deferred**: `GenericWildernessSpotAttendable` references `ItemDefinition` assets directly via serialized `ItemYield` fields, and `InventoryManager` only needs `ItemDefinition` identity (not id-based lookup). Revisit if a system needs to resolve items by string id (e.g. save/load, crafting recipes).
- **`InventoryManager.AddItem` returns the number of units actually added**; any remainder when both the matching stacks are full and `CarryLimit` (8 stacks) is reached is refused outright rather than dropped on the ground or queued — callers (e.g. `GenericWildernessSpotAttendable`) log a "no room to carry it" message when `added == 0`.
- **`InventoryUI` reads `InventoryManager.Instance` in `Start()`, subscribing to `InventoryChanged` there** rather than `OnEnable()` — same cross-GameObject `Awake`-ordering reasoning as `AttentionManager`'s `Start()`-time lookups, since `Start()` is guaranteed to run after every `Awake()`.
- **"Inventory UI" GameObject** (`UIDocument` + `InventoryUI`) is another code-first overlay sibling to "Overlay UI", anchored top-left vs. the attendable overlay's placement — both follow the same runtime-`PanelSettings`/`VisualElement`-tree-in-code pattern established in Iteration 3.
- **New `Mossmark.Day` namespace (Iteration 5)** holds the daylight/day-clock system — `DayCycleManager`, `DayCycleAmbientTextData`, `DayCycleUI`. Kept separate from `Mossmark.Attention` even though attention spends daylight: "attention is the clock" per PROTOTYPE2.md, but the day-cycle state itself is a concern other future attendables (buildings, NPCs, the Iteration 6 bedroll) will also read/write without being part of the attention framework.
- **`IAttendable.RequiresDaylight`** is the new interface member deciding whether `AttentionManager` calls `DayCycleManager.SpendDaylight()` after `OnAttentionComplete()`, and whether `AttendableOverlayUI` replaces the attendable's normal interaction line with "Too late to start that now." when `DayCycleManager.HasDaylight` is false. `AttentionManager.HandleHoldStarted` also checks this *before* starting a hold, so a zero-daylight attempt on a daylight-costing target consumes nothing and never enters `Attending`. `GenericWildernessSpotAttendable.RequiresDaylight => true`; non-daylight attendables (e.g. the Iteration 6 bedroll) will return `false`.
- **`DayCycleManager.SpendDaylight()` costs a flat 1 per completed daylight-costing attention** — the direct successor to P1's `ActionsPerDay`/`ActionsRemaining` action-pool. `MaxDaylight` defaults to 24, split into 6 even `DayPhase` bands (Dawn → Morning → Midday → Afternoon → Evening → Dusk, 4 daylight each) via a ratio-threshold `GetPhase()`, so phase-crossing ambient text fires 5 times over a full day.
- **`DayCycleAmbientTextData` is a `ScriptableObject`** (`Assets/Game/Data/Day/DefaultAmbientText.asset`) holding one or more candidate lines per `DayPhase`; `DayCycleManager` picks one at random on each phase crossing and logs it via `Debug.Log`, following Iteration 4's "no toast/notification UI yet" precedent rather than adding a transient-message UI.
- **Wiring `DayCycleManager.ambientTextData`** (a `ScriptableObject` reference, unlike the code-first UI components) couldn't be set via MCP-Unity's `update_component` per the asset-reference limitation noted above — the GameObject/component were added via MCP, then the scene `.unity` file was hand-edited to add `{fileID: 11400000, guid: ..., type: 2}` for the `DefaultAmbientText` asset, followed by `load_scene` to pick up the change before re-saving.
- **"Day Cycle UI" GameObject** (`UIDocument` + `DayCycleUI`) is another code-first overlay sibling, anchored top-right with a phase label and a `[####....]`-style daylight bar (20 segments, same text-bar pattern as the hold-progress bar) that dims to translucent white at zero daylight. (Superseded — the daylight bar was replaced by an ambient-text label.)
- **`DayCycleManager.Rest()` (Iteration 6)** is a coroutine-driven fade-to-black / reset / fade-in: `RestRoutine()` sets `IsTransitioning = true`, fades `FadeAmount01` 0→1 over `transitionFadeDuration` (0.6s), resets `DaylightRemaining` to `maxDaylight` and calls `SetPhase(DayPhase.Dawn)`, fades 1→0, then clears `IsTransitioning`. `SpendDaylight`'s inline phase-change/ambient-text logic was extracted into the shared `SetPhase()` helper so `Rest()` reuses the same ambient-text-on-phase-change behavior.
- **`DayCycleManager.IsTransitioning` is the cross-system lock for the rest transition**: `PlayerController.HandleInput()` zeroes `movement` and returns early while it's true, and `AttentionManager.HandleHoldStarted()` refuses to start a new hold — generalizing P1's `RestInteractable` lock to the whole game loop (movement + attention) rather than just the bedroll itself.
- **`BedrollAttendable` lives in `Mossmark.Day`**, not `Mossmark.World` or `Mossmark.Attention` — it's tightly coupled to `DayCycleManager.Rest()` rather than being a generic world/wilderness attendable. It sets `RequiresDaylight => false` (resting restores daylight, so the "too late to start that now" zero-daylight gate doesn't apply) and `CanAttend() => true`, since `AttentionManager.HandleHoldStarted`'s `IsTransitioning` check already covers re-attending the bedroll mid-transition.
- **`DayTransitionFadeUI`** is a code-first overlay sibling like the others, but with `UIDocument.sortingOrder = 100` so its full-screen black `VisualElement` renders above Overlay/Inventory/Day Cycle UI. It polls `DayCycleManager.Instance.FadeAmount01` every frame and sets the overlay's `backgroundColor` alpha directly — same "manager owns progress, UI just displays it" pattern as `AttendableOverlayUI`'s hold-progress bar, applied to a continuous fade instead of a discrete bar.
- **"Reseeding wilderness spots" (third clause of the Iteration 6 deliverable) is currently a no-op**: `GenericWildernessSpotAttendable` is stateless and instant-yield ("no state, no wait" per PROTOTYPE2.md), so attending Field/Clay Pit again after `Rest()` already behaves identically to before — there's nothing to reseed yet. This will need a real `OnDayAdvanced`-style hook once Iteration 7 (Tended-Style Spots) introduces per-spot state.
- **`DayCycleManager.DayAdvanced` event (Iteration 7)** fires once per `Rest()`, in `RestRoutine()` right after `DaylightRemaining`/`CurrentPhase` are reset to Dawn but before the fade back in. `TendedSpotAttendable` subscribes in `Start()` and unsubscribes in `OnDestroy()` to advance its mark → wait → harvest countdown.
- **`TendedSpotAttendable` (Iteration 7) lives in `Mossmark.World`**, alongside `GenericWildernessSpotAttendable` — both are wilderness-spot attendable types; tended spots add an `Unmarked`/`Marked`/`Ready` state machine (ported from P1's `TendedSpotInteractable`) on top of the same `IAttendable` shape.
- **State-driven `CanAttend()`/overlay text (Iteration 7)**: `Unmarked` → `CanMark()` (gated by the cap; overlay shows "(too many spots already tended)" without consuming a hold when capped), `Marked` → `false` (overlay shows "ready in N rest(s)" countdown, blocked from re-marking), `Ready` → `true` (harvest allowed). `Mark()` and `Harvest()` both run inside a single `OnAttentionComplete()`, switched on `state`.
- **`MaxConcurrentMarked` cap via a `static Dictionary<string, int>` keyed by `displayName`** — generalizes P1's `TendedSpotConfig`-keyed `_markedCounts` (one shared SO per spot type) to P2's per-component-config approach, where each `TendedSpotAttendable` instance carries its own `displayName`/`maxConcurrentMarked` rather than referencing a shared asset. `OnDestroy()` decrements the count if a `Marked` spot is destroyed.
- **`TendedSpotAttendable.RequiresDaylight => false`, for both marking and harvesting**: `AttentionManager.CompleteAttention` reads `RequiresDaylight` *after* `OnAttentionComplete()` runs, by which point a successful `Harvest()` has already reset `state` back to `Unmarked` — a state-dependent answer at that point can't distinguish "just harvested" from "nothing to harvest." Rather than add latching state to work around this read-order, tended spots are framed as free "tending" check-ins (cultivation), distinct from `GenericWildernessSpotAttendable`'s daylight-costing active extraction.
- **Visual state feedback via `SpriteRenderer.color` tinting**, not sprite/material swaps: `TriangleSpriteGenerator` bakes its base color into the sprite texture at `Awake()`, and `TendedSpotAttendable.UpdateVisual()` sets the (default-white, multiplicative) `SpriteRenderer.color` directly — unchanged (white) for Unmarked, dark grey for Marked, warm yellow for Ready.
- **`Assets/Refresh` needed after creating a brand-new component's `.cs` file before MCP-Unity's `update_component` can find it**: `recompile_scripts` compiled `TendedSpotAttendable` with 0 errors, but `update_component` still failed with "Component type not found" because no `.meta` existed yet for the new file — `recompile_scripts` compiles but doesn't trigger an `AssetDatabase` import. `execute_menu_item` with `menuPath: "Assets/Refresh"` triggers the import (the `.meta` then appears), after which `update_component` succeeds.
- **New `Mossmark.Development` namespace (Iteration 8)** holds the generic dependency/response resolver: `IDependencyCondition` and its four implementations (`ItemAvailableCondition`, `EntityStateCondition`, `WorldStateCondition`, `TimeCondition`), `DevelopmentStage`, `DevelopmentTrack`, `DevelopableEntity`, and the static `WorldState` flag registry. This generalizes P1's `Entity`/`UpgradeDependency`/`UpgradePool`/`TownEntity` per PROTOTYPE2.md Section 4.
- **`IDependencyCondition` implementations cover four condition shapes**: `ItemAvailableCondition(ItemDefinition item, int count)` checks `InventoryManager.Instance.GetQuantity(item) >= count` (the new `InventoryManager.GetQuantity()` method); `EntityStateCondition(DevelopableEntity entity, int minStageIndex)` checks another entity's `CurrentStageIndex`; `WorldStateCondition(string flagId, bool expected, string unmetDescription)` checks `WorldState.GetFlag(flagId)`; `TimeCondition(int attentionsRequired)` is always "satisfied" in the sense that it never blocks progress accumulation — it exists purely to make `DevelopmentStage.ProgressCost` meaningful. Each condition exposes `IsSatisfied()` and a human-readable `Describe()` for the "needs ..." overlay/log text.
- **`DevelopmentStage`** is a plain C# class: `Id`, `DisplayName`, `ProgressCost`, and a `params IDependencyCondition[] dependencies` array. `DevelopmentTrack` wraps a flat ordered list of stages (no branching/tree — "Tried not chosen" per PROTOTYPE2.md: stages apply automatically once eligible, not via player choice).
- **`DevelopableEntity` is an abstract `MonoBehaviour` base class**, not a `ScriptableObject` — `PendingProgress`/`CurrentStageIndex` are per-instance runtime state, so each hand-placed entity (Old Cairn, Watcher's Post) subclasses it directly and constructs its own `DevelopmentTrack` in `Awake()`. **Tracks are built programmatically in code, not via `[SerializeReference]` polymorphic lists** — MCP-Unity's `update_component` has no way to author polymorphic `IDependencyCondition` arrays through the inspector, so each subclass's `Awake()` hardcodes its `DevelopmentTrack(new DevelopmentStage(...))` construction, with only the leaf data (e.g. the `sticks` `ItemDefinition`, the `oldCairn` cross-reference) as serialized fields.
- **`ResolveAttention()`'s 3-step order resolves the `TimeCondition` chicken-and-egg problem**: (1) `AddProgress(1)` unconditionally increments `PendingProgress`, regardless of whether the current stage's dependencies are satisfied; (2) `TryApplyStage()` then checks `PendingProgress >= stage.ProgressCost` AND all `dependencies` satisfied — if both hold, it applies the stage; (3) if the stage didn't apply, `LogDependencyReport()`/`GetNeedsOrDefault()` report which conditions are still unsatisfied. This lets a `TimeCondition(2)` stage accumulate progress even while another condition remains unsatisfied — progress and dependency-gating are decoupled.
- **`RequiresDaylight => LastAttentionConsumedDaylight` (the "read-after-`OnAttentionComplete`" pattern, first established by `TendedSpotAttendable` in Iteration 7)**: `AttentionManager.CompleteAttention` calls `OnAttentionComplete()` (which runs `ResolveAttention()` and sets `LastAttentionConsumedDaylight`) *before* reading `RequiresDaylight` — so a blocked/no-progress-applied attend consumes no daylight, while a stage-applying attend does.
- **`InventoryManager.GetQuantity(ItemDefinition item)`** returns the total carried count across all stacks of that item — added to back `ItemAvailableCondition` without exposing the internal stack list.
- **`WorldState`** is a static class wrapping a `Dictionary<string, bool>` flag registry (`GetFlag`/`SetFlag`), scoped process-wide for this prototype (no save/load yet).
- **Three hand-placed Iteration 8 test entities**: "Old Cairn" (`OldCairnAttendable`, grey, world (3,3)), "Signal Fire" (`SignalFireAttendable`, orange, world (3,-3)), "Watcher's Post" (`WatchersPostAttendable`, slate blue, world (-3,-3)). All three replaced by `LandmarkAttendable` in Iteration 24 (G8).
- **Deferred from Iteration 8**: a `TownEntity`/town-scope registry for conditions that need town-wide state; `VisualColor`/yield-pool fields on `DevelopmentStage` for stage-driven visual changes. Revisit when a later iteration needs town-level aggregation or stage-driven art changes.
- **`IAttendable.ContinueAttending` (Iteration 8.5)** is the new interface member, read *after* `OnAttentionComplete()` like `RequiresDaylight` — true means the hold resets and starts another tick, false means the hold ends here regardless of whether the attend key is still held.
- **`AttentionManager`'s `Attending` state is now a loop (Iteration 8.5)**: a new `isHeld` field tracks the raw Attend-button state from `AttentionInput`'s `HoldStarted`/`HoldReleased`. `CompleteAttention()` calls `OnAttentionComplete()`, spends daylight if `RequiresDaylight`, then calls `ShouldContinue()` — which requires `isHeld`, `target.ContinueAttending`, `detector.IsInRange(target)`, and (only if this tick spent daylight) `DayCycleManager.HasDaylight`. If all hold, the tick loop resets and continues; otherwise `FinishAttending()` runs as before.
- **`DevelopableEntity.LastAttentionConsumedDaylight` split into `LastAttentionMadeProgress` + `LastAttentionAppliedStage` (Iteration 9)**: `LastAttentionMadeProgress` is true whenever this tick's dependencies were satisfied; `LastAttentionAppliedStage` is true only on the tick that actually crosses the stage threshold.
- **`InventoryManager.RemoveItem(ItemDefinition item, int quantity)` (Iteration 9)** mirrors `AddItem`'s "returns units actually affected" shape.
- **`BuildingAttendable` (`Mossmark.Development`, Iteration 9)** is a generic, reusable `DevelopableEntity` subclass — "Buildings" is a whole taxonomy category that later world-gen will spawn many instances of with different configs. All building-specific data is serialized fields; the single-stage `DevelopmentTrack` is built in `Awake()` per the no-inspector-polymorphism constraint.
- **`TimeCondition.IsSatisfied()` now always returns `true` (Iteration 10)**, restoring its original Iteration 8 intent — the threshold check belongs to `DevelopmentStage.ProgressCost` via `TryApplyStage()`, not the dependency itself.
- **`NpcAttendable` (`Mossmark.Development`, Iteration 10)** is the first entity where `TryApplyStage()`'s random-among-available selection acts as an actual specialization draw. Its `DevelopmentTrack` holds three same-cost stages (`forager`/`caretaker`/`tinkerer`). `CanAttend() => CurrentStageIndex < 0` makes the NPC inert once specialized.
- **`DeclaredSpecializationNeeds` (`Mossmark.Development`, Iteration 11)** is a static registry — a `HashSet<string>` of specialization ids (`Declare`/`Contains`/`Consume`) — fulfilling the "Building → NPC demand" shared-needs registry.
- **`DevelopableEntity.TryApplyStage()` gained a priority-filter step (Iteration 11)**: after computing `available`, if any candidate's `Id` matches a currently-declared need, `available` narrows to just those candidates before the random pick.
- **`PlaceArchetype` and `RegionData` (`Mossmark.World`, Iteration 12)** are new ScriptableObjects under `Assets/Game/Data/World/`. `PlaceArchetype` bundles a wilderness-spot definition with a specialization definition — one asset is "everything a session draws when it picks this place."
- **`WorldGenerator` (`Mossmark.World`, Iteration 12) uses `[DefaultExecutionOrder(-1000)]`** so its `Awake()` runs before every other scene object's `Awake()` — critically, before the hand-placed Wanderer's `NpcAttendable.Awake()`, which reads the static `WorldGenerator.SelectedArchetypes` while building its `DevelopmentTrack`.
- **`DefaultRegion` (`Assets/Game/Data/World/DefaultRegion.asset`, Iteration 12)** ships a pool of 5 `PlaceArchetype` assets (`archetypeSelectionCount: 3`, so each session selects 3 of 5): **Fen Bog** (→ Bog Keeper), **Old Road** (→ Herald), **Deep Wood Shrine** (→ Hedge Witch), **Reed Marsh** (→ Weaver), **Old Quarry** (→ Stonemason).
- **No Play-mode automation tool exists in MCP-Unity (Iteration 12)**: `execute_menu_item` with `"Edit/Play"` fails. Verification is via `recompile_scripts`, `get_console_logs`, `get_gameobject`, and manual playtest sequences.
- **`AttentionDuration` changed from `0f` to `2f` (pre-Iteration-13 polish pass)** for Old Cairn, Signal Fire, Bramble Patch's mark and harvest, and Watcher's Post — these one-shot checks now require a 2-second hold like the ongoing attendables.
- **`GenericWildernessSpotAttendable`'s `tickInterval` (single field, Iteration 8.5) replaced with `minTickInterval`/`maxTickInterval` (defaults `1.5f`/`2f`) (pre-Iteration-13 polish pass)** — `AttentionDuration` now returns a `currentTickInterval` rerolled by `RollTickInterval()` on each tick.
- **`DevelopableEntity.CanMakeProgress()` (pre-Iteration-13 polish pass)** — a new public method (`GetNextStage() != null && stage.AreDependenciesSatisfied(this)`) now exposed so `CanAttend()` can gate on it *before* entering the `Attending` hold.
- **"stamina" renamed to "daylight" project-wide (pre-Iteration-13 polish pass)**: all identifiers, comments, and scene serialized fields updated. No behavioral change.
- **`DayCycleUI`'s daylight bar replaced with an ambient-text label (pre-Iteration-13 polish pass)**: `DayCycleManager` gained `public event Action<string> AmbientTextChanged`, fired from `SetPhase()`. `DayCycleUI` now subscribes to it and displays the per-phase ambient text lines in the HUD.
- **`RealizedSpecializations` (`Mossmark.Development`, Iteration 13)** mirrors `DeclaredSpecializationNeeds`: a static `HashSet<string>` of specialization ids that *have been realized* by some NPC. The new `SpecializationRealizedCondition` checks `RealizedSpecializations.Contains(specializationId)`.
- **`ItemYieldRoller` (`Mossmark.World`, Iteration 13)** extracts `GenericWildernessSpotAttendable`'s yield-roll logic into a static helper `Roll()`, shared by `GenericWildernessSpotAttendable` and the new `PoiAttendable`.
- **`PoiAttendable` (`Mossmark.World`, Iteration 13)** is a new `IAttendable`, deliberately not a `DevelopableEntity` — a POI has nothing to develop, only to unlock. Mechanically it's an ongoing wilderness spot plus an optional `IDependencyCondition gate`.
- **`PlaceArchetype` gained a `[Header("Point of Interest")]` block (Iteration 13)** bundling all POI data alongside the existing spot/specialization data. Each POI unlocks once any NPC realizes that archetype's specialization.
- **Five archetypes' POI data (Iteration 13)**, each pairing the archetype's *former rare wilderness yield* as the POI's new common yield with a POI-exclusive rare item: Fen Bog Hollow (Crow Feather / Sunken Bell), Old Road Checkpoint (Old Coin / Tally Stick), Old Quarry Cut (Quartz / Carved Sigil Stone), Reed Marsh Blind (Duck Egg / Woven Reed Charm), Deep Wood Shrine - Inner Grove (Raven's Eye / Antler Talisman).
- **Hedge Witch + Bog Keeper post-spec stages (Iteration 17)**: each archetype provides two stages using its own `CommonYields[0].Item` and `RareYield.Item` as item dependencies, gated by `SpecializationRealizedCondition + ItemAvailableCondition`. Stage ids double as WorldState flag keys — `HandleDeveloped` calls `WorldState.SetFlag(stage.Id, true)` for any non-pool stage. Hedge Witch: "hedge_witch_wound_lore" (progressCost 3, 2x Mistletoe) and "hedge_witch_ravens_eye" (progressCost 4, 2x Raven's Eye). Bog Keeper: "bog_keeper_drainage" (progressCost 3, 3x Bog Iron) and "bog_keeper_iron_sense" (progressCost 4, 2x Crow Feather).
- **`OutcomeRequest.DaylightCostMultiplier` (Iteration 17)** is the second field added to `OutcomeRequest`. `WoundLoreModifier` checks `WorldContext.GetFlag("hedge_witch_wound_lore")` and sets `DaylightCostMultiplier = 0f` — eliminating the extra `badDaylightCost` daylight spend on bad wandering-thing outcomes.
- **`WorldStateChanceModifier(flagId, multiplier)` (Iteration 17)**: a generic companion to `WoundLoreModifier` for flag-based probability biasing. Wired for Herald post-spec flags (`herald_trail_markers × 1.3`, `herald_toll_records × 1.2`).
- **Content Pass (post-Iteration-17): deeper development loop** — `BuildingAttendable` gained an optional Stage 2; `PlaceArchetype` gained a `[Header("Building")]` block; `WorldGenerator` replaced single `buildingMaterial`/`buildingPosition` with `buildingPositions: Vector2[]`; `NpcAttendable` universal pool narrowed to forager/caretaker/tinkerer only (no hardcoded "smith"), `progressCost` default raised from 4 → 8; Herald post-spec stages added; all 5 archetype assets updated; hardcoded Smithy removed; Roughhand NPC added; intended arc per session: Stage 1 revival → NPC specialization + POI opens → NPC post-spec stage 1 → Stage 2 → NPC post-spec stage 2.
- **`WildernessSpotDefinition` ScriptableObject (Iteration 18)** bundles all per-type data for either a generic or tended spot in a single `[CreateAssetMenu]` asset. The `kind` enum (`Generic`/`Tended`) drives which component `SpawnSpotFromDefinition` adds.
- **Rejection-sampling position placement (Iteration 18)**: `WorldGenerator.FindValidPosition()` draws up to 200 candidate points uniformly within `WorldLayoutGenerator.WildernessBounds`, rejects points inside `TownBounds` or within `minSeparation` (2 units) of any already-placed object, and falls back to the closest-so-far candidate after 200 attempts.
- **Seven spot types (Iteration 18)**: Field, Clay Pit, Mushroom Patch, Stone Outcrop, Hollow Log (all generic ongoing), Bramble Patch (tend, 1 rest), Bee Skep (tend, 2 rests). Six new `ItemDefinition` assets: `mushrooms`, `spore_cluster`, `pebbles`, `bark_strips`, `beeswax`, `honeycomb`.
- **`WanderingThingSpawner` random placement (post-Iteration-18 polish)**: hardcoded `spawnPositions` removed; `Spawn()` now calls `FindSpawnPosition()`, rejection-sampling within `WorldLayoutGenerator.WildernessBounds` with `minDistFromTown` exclusion zone.
- **`WanderingThingAttendable` outcome notifications (post-Iteration-18 polish)**: `OnAttentionComplete()` calls `NotificationManager.Post()` for both good and bad outcome branches.
- **Rare-drop interrupt intentionally removed from `GenericWildernessSpotAttendable` and `PoiAttendable`**: `continueAttending = !RollYield()` was removed once `NotificationUI` landed — the banner already surfaces the rare drop, making the hold-ending interrupt redundant. `ItemYieldRoller.Roll()`'s `bool` return value and the `continueAttending` field are kept as mechanisms for future interrupt conditions.
- **`BuildingStageDef` (`Mossmark.Development`, Iteration 21)** is a `[Serializable]` class — one entry per stage in a building's development path. `BuildingAttendable` replaces its parallel stage field groups with `BuildingStageDef[] stages`; `PlaceArchetype` replaces its mirroring flat fields with `buildingDilapidatedName`, `buildingDilapidatedColor`, and `BuildingStageDef[] buildingStages`.
